# ADR 0003 — Crawlable detail pages via Pages Functions HTML rewrite

- **Status:** Accepted (implemented 2026-07-17; deploy pending manual `pnpm deploy`)
- **Date:** 2026-07-17
- **Builds on:** ADR 0001 (Vite SPA removed default SSR)

## Context

After the Vite SPA migration (ADR 0001), `/anime/$malId` and `/manga/$malId`
were SPA shells — every crawler saw the same empty `index.html` for thousands
of long-tail "«title» anime" queries. A fleet GEO/SEO audit identified this
as the largest raw pSEO unlock in the fleet. The repo already held the full
dataset locally and the site was already on Cloudflare Pages, so unique
indexable HTML per title could cost no build step and no runtime API calls.

## Decision

Intercept detail routes with **Pages Functions**
(`functions/anime/[malId].ts`, `functions/manga/[malId].ts`) that rewrite the
SPA shell before serving:

- unique `<title>`, meta description, canonical, OG/Twitter tags;
- JSON-LD (`TVSeries`/`Movie` for anime by type, `Book` for manga);
- a `<div hidden data-ssr>` summary (h1 + synopsis + facts table) so the page
  has real text content;
- unknown/invalid IDs → shell with `<meta name="robots" content="noindex">`.

A compact SEO dataset is generated at **build time** by
`scripts/build-seo-dataset.mjs` (prebuild) from the cleaned JSON, filtered to
5,306 anime (members ≥ 20k) + 2,288 manga (members ≥ 10k), emitting
`src/data/seo-{anime,manga}.json`. Chunked sitemaps are generated by
`scripts/build-sitemaps.mjs` (postbuild) into `dist/`.

The rewrite itself is a **pure function** in `src/seoRewrite.ts` (string
replacement over `<!-- seo:start/end -->` markers in `index.html`) so it is
deterministic and testable in vitest without workerd. HTML escaping and
`</script>` injection protection are covered by unit tests; an e2e test
asserts the app still mounts after rewrite.

## Why Pages Functions and not the Worker

The SPA is deployed on Cloudflare Pages; Pages Functions are the native
server-side layer in front of the SPA catch-all. The `mal-api` worker serves
JSON and is not in the HTML path.

## Consequences

- Positive: thousands of unique, indexable detail pages with no SSR framework
  and no runtime API calls.
- Constraint: only titles above the popularity threshold get crawlable HTML;
  below-threshold IDs get `noindex`. The threshold is tunable in
  `scripts/build-seo-dataset.mjs`.
- Constraint: the SEO dataset is committed and must regenerate cleanly — CI
  should assert `git diff` is empty after `prebuild` (currently a manual
  expectation; see `STATUS.md` follow-ups).
- Risk: duplicate content vs MAL — synopsis excerpts are capped at 300 chars,
  the page leads with our own facts markup, and canonical stays on our host.
- Risk: HTML injection must not break SPA hydration — injected tags live in
  `<head>` + one hidden SSR block; e2e asserts the app still mounts.
